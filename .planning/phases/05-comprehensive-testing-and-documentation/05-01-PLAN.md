
   - A marketplace script input holding the given UTxO value with the datum
   - The given outputs and signers
   - Uses `mkTxInfo` (always valid range)
   - Wraps in `mkSpendingCtx` with MktBuy/MktWithdraw redeemer

**AttackScenarios.hs additions:**

1. Add imports:
   - `import qualified Carbonica.Validators.Marketplace as Marketplace`
   - `import Carbonica.Validators.Marketplace (MarketplaceDatum(..), MarketplaceRedeemer(..), Wallet(..))`
   - `import Test.Carbonica.TestHelpers (mkMarketplaceCtx, testRoyaltyAddr, lovelaceSingleton)` (add to existing import)

2. Update `attackScenarioTests` to include `med01Tests, med02Tests, med03Tests, med04Tests` in the list.

3. Add `med01Tests` group (MED-01: Marketplace UTxO token verification):
   - `med01a_buyWithEmptyUtxo`: MarketplaceDatum claims 100 COT but UTxO holds only ADA. Use `testAttackRejected3` with `Marketplace.untypedValidator`, first arg `toBuiltinData testIdNftPolicy`, second arg `toBuiltinData testRoyaltyAddr`. Expected rejection MKE007.
   - `med01b_buyWithWrongToken`: UTxO holds wrong CurrencySymbol COT. Expected rejection MKE007.
   - `med01_positive_buyWithCorrectCot`: UTxO holds correct COT tokens, correct payouts to seller + royalty + buyer. Use `testAttackAccepted3`.

4. Add `med02Tests` group (MED-02: Zero/negative price):
   - `med02a_zeroPriceBuy`: datum with `mdAmount = 0`. Expected rejection MKE008.
   - `med02b_negativePriceBuy`: datum with `mdAmount = -1`. Expected rejection MKE008.
   - `med02_positive_positivePriceBuy`: datum with valid positive price, correct outputs. Use `testAttackAccepted3`.

5. Add `med03Tests` group (MED-03: Royalty rounding evasion):
   - `med03a_royaltyRoundingFloor`: price = 1 lovelace, attacker omits platform payment entirely (no output to royaltyAddr). `P.max 1 ((1 * 5) `div` 100)` = 1 lovelace minimum. Expected rejection MKE004 (platform not paid).
   - `med03_positive_royaltyFloorPaid`: price = 1, but outputs include 1 lovelace to royaltyAddr. Use `testAttackAccepted3`.

6. Add `med04Tests` group (MED-04: DaoGovernance vote non-vote field mutation):
   - Build `mkDaoVoteSpendCtx` local helper that creates a DaoGovernance vote spending context with given input/output GovernanceDatum and signers. Use `mkTxInfo` (always range) since vote needs to be before deadline.
   - `med04a_submitterMutated`: input gdSubmittedBy = alice, output gdSubmittedBy = eve (all other fields correct). Expected rejection DGE019.
   - `med04b_actionMutated`: input gdAction = ActionUpdateFeeAmount 200M, output gdAction = ActionAddSigner dave. Expected rejection DGE020.
   - `med04c_deadlineMutated`: input gdDeadline = oneWeekMs + 1M, output gdDeadline = oneWeekMs + 9M. Expected rejection DGE021.
   - `med04_positive_validVotePreservesFields`: all non-vote fields preserved, vote count incremented correctly. Use `testAttackAccepted2`.

**IMPORTANT:** Marketplace's `untypedValidator` second argument is `royaltyAddr :: PubKeyHash` (NOT a CurrencySymbol). Always use `toBuiltinData testRoyaltyAddr` as the second arg. For MED-04, DaoGovernance.untypedSpendValidator is 2-arg (idNftPolicy -> ctx), so use `testAttackRejected2`/`testAttackAccepted2`.
