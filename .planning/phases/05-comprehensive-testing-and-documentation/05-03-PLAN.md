  <action>
For each of the 5 Types/ modules, add Haddock documentation following the established style:

**Haddock comment format (from existing Core.hs patterns):**
```haskell
-- | Purpose of the function in one sentence.
--
-- Additional detail about parameters if needed.
{-# INLINEABLE functionName #-}
functionName :: Type -> Type
```
The Haddock comment goes BEFORE the INLINEABLE pragma (which goes before the type signature). This matches the existing pattern in Core.hs.

**For record fields, use `-- ^` syntax:**
```haskell
data MyType = MyType
  { field1 :: Type  -- ^ Description of field1
  , field2 :: Type  -- ^ Description of field2
  }
```

**Module-level headers:** Ensure each module has a proper `{- | ... -}` header. If one exists, expand it. If missing, add one with Module, Description, and License fields.

**Core.hs:** Some Haddock already exists (e.g., mkLovelace has examples). Add/expand for:
- All newtypes: `Lovelace`, `CotAmount`, `CetAmount`, `Percentage` (document what they represent)
- All smart constructors: `mkLovelace`, `mkCotAmount`, `mkCetAmount`, `mkPercentage` (document constraints)
- All accessors: `lovelaceValue`, `cotValue`, `cetValue` (document what they return)
- All address newtypes: `DeveloperAddress`, `FeeAddress`, `ValidatorAddress`
- All address accessors: `toPubKeyHash`, `developerToPkh`, `feeToPkh`, `validatorToPkh`
- Error types: `QuantityError` and its constructors
- Do NOT add Haddock to TH-generated code (makeIsDataSchemaIndexed, makeLift lines)

**Config.hs:** Add/expand for:
- `Multisig` type and its fields (`msRequired`, `msSigners`)
- `ConfigDatum` type description (note: constructor unexported, use `mkConfigDatum`)

**Emission.hs:** Add/expand for:
